What is the Dom you ask? Good question my foolish young coder. DOM is short for Document Object Model, which refers to just about everything on an HTML page. It is fashioned like a watersystem, begining with the Document, flowing straight down to the HTML tag before splitting into head and body, which then break off into all the different bits of code which are included in the HTML page(i.e. the HTML code). Pull up a full page of HTML code and you can chart the course that the DOM takes, noting that each end tag denote the finality of that perticular sections scope. These objects are called nodes. These nodes(Objects in HTML) are usually run on servers rather than on the machiene that is trying to view them. In this model the client machiene (phone, laptop, tv at this point) sends off a request to view a page. This request is recieved by a domain, which then attempts to fufill that request. This request is directed to a server, which is a bunch of computers wherein the script for the website is stored. If the server finds the requested script it runs it, creating a webpage out of HTML elements and whatever else the script is comprised of. This finished product is then sent back to the client for thier viewing pleasure. The chiral of this is the client side model. In the client side model the unfinished data is passed directly to the client, and the clients browser does the work of constructiing it. This is esssentially a static model, in the way described. Websites made by either method would not have any changes made to them. They either exsist or they do not. Making these websites fluid recuires events. An event is anything that causes a change to that static enviorment. User input is one common sort of event, but there are others, such as portions of a website that perform actions or display movement. Event listeners are used to set up scenarios in which user input can trigger events. Some common examples of this would be a button that performs an action, or a section that maginifies when hovered over. To be clear this is not the extent of what event listners can be programmed to do nor the entirery of events to which event listeners can be programmed to respond for, they are simply common examples. Window.event is a method that can be called to take a snapshot of what is going on at the exact time that a event trigger goes off. Window is the true root function of the dom, behind even document, so taking a snapshot of it provides the full state of the website, from user input, to what functions are active to even the users mouse position. As you have hopefully already learned, the greater the amount of information whilst programing, the more acurate the problem solving or the more creative the code that can be written.
